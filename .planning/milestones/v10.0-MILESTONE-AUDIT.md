### Dimension 2 — Schema references (111 WARNINGs)

111 WARNINGs all from the Levenshtein close-match check. Two categories:

1. **camelCase ↔ snake_case confusion (98 WARNINGs):** Design docs use
   snake_case in narrative prose (e.g., `opened_at`, `closed_at`,
   `current_round`, `max_rounds`, `state`) while the canonical schema
   fields are camelCase JSON property names (`openedAt`, `closedAt`,
   `currentRound`, `maxRounds`, `status`). The lint correctly flags the
   close match — the snake_case form is a typo OR a narrative variant
   that should be canonicalized. For v10.0 these are accepted as narrative
   shorthand; v11.0 PoC implementer should be aware that the canonical
   forms are camelCase per `round-table-state-schema.yaml`.

2. **Compound-name drift (13 WARNINGs):** e.g., `core_memory` /
   `working_memory` / `archival_memory` are tier names from §2.6.6 —
   they are NOT in the canonical schema (which has `memory_scope` enum
   + `memory.max_records` nested). These are conceptual tier names,
   not strict schema fields. Accepted as design-doc vocabulary.

**Sample findings:**

**Interpretation:** All 111 WARNINGs are heuristic noise from Levenshtein
