# v7.0 openclaw → hermes-agent Migration Report

This is the canonical close-out artifact for milestone v7.0 — the migration of openclaw's primary-agent capabilities (skills + identity + memory) into hermes-agent, so that hermes-agent can take over the primary-agent role with capability parity. v7.0 is the first milestone to operate outside the `skills/movie-experts/` subsystem; it broadens PROJECT.md from "Movie-Experts Suite" to "Hermes Agent — Kai's Personal Agent Platform".

**Milestone purpose:** Prior to v7.0, Kai operated two parallel agents — `openclaw` (the primary day-to-day agent with coding-delegation, AIGC routing, and memory-of-record) and `hermes-agent` (initially a movie-experts specialist). v7.0 consolidates the primary-agent capabilities into hermes-agent so that openclaw can eventually be deprecated. The migration is intentionally narrow: only the 3 highest-leverage capability categories (skills / identity / memory) move in v7.0; configuration, runtime state, and 5 lower-priority skill families are explicitly deferred (see §Explicitly Skipped Items).

## Executive Summary

v7.0 shipped the **structural migration** of openclaw's primary-agent capabilities into hermes-agent across 3 implementation phases (34, 35, 36) plus 1 validation phase (37). Live runtime confirmation (tmux spawn, mem0 ingestion, SOUL routing observation) is handed off to the operator per scoped-boundary decisions in Phase 37 CONTEXT.md.

- **Skills (Phase 34):** 2 skills migrated — `coding-agent` (unified tmux-backed delegation across Codex / Claude Code / Pi / OpenCode) and `tmux-agents` (background session lifecycle: spawn / list / attach / get-results) — both at `skills/autonomous-ai-agents/` with hermes-native frontmatter. Coexistence decision documented as SUPPLEMENT (not replace); bidirectional `related_skills` graph wired across all 6 autonomous-ai-agents skills.
- **Identity (Phase 35):** `~/.hermes/SOUL.md` enhanced non-destructively with openclaw's 4 AIGC routing categories (immediate-execution / cognitive / expert-management / default), adapted from openclaw's regex+MCP topology to hermes' single-agent + skill-discovery + mem0 + curator topology. Original 515-byte Hermes Nous-Research identity preserved byte-for-byte as Section A.
- **Memory (Phase 36):** Memory ingestion tooling built — `batch_ingest.py` (idempotent SHA-256 dedup) + `spot_check.py` (5-query semantic-search CLI) — at `plugins/memory/mem0/scripts/`. USER.md migrated to `~/.hermes/memories/USER.md` with openclaw-origin frontmatter. Dry-run enumerates 124 files (~817KB) successfully; live ingestion deferred to operator pending `MEM0_API_KEY` configuration.
- **Coexistence (Phase 34):** `COEXISTENCE.md` is the canonical SUPPLEMENT decision artifact; capability matrix shows the 6 skills partition cleanly along 4 dimensions.
- **Migration report (this document, Phase 37):** Canonical close-out artifact documenting every transform decision and every explicitly skipped item with one-line rationale — load-bearing for preventing v8.0 scope-creep reversal.

**Closing statement:** v7.0 ships the structural migration; live runtime verification (tmux spawn + mem0 ingestion + SOUL routing observation) is handed off to operator per scoped-boundary decisions in Phase 37 CONTEXT.md. The structural preconditions are fully verified in `37-01-BENCHMARK-RESULTS.md`; runtime confirmation is documented in §Operator Action Items below.

**Why this milestone matters:** Prior to v7.0, Kai maintained two parallel agent runtimes (openclaw as the primary day-to-day driver + hermes-agent as a movie-experts specialist). v7.0 is the consolidation milestone — once the operator smoke-tests pass, hermes-agent can take over the primary-agent role for the migrated capabilities, and openclaw's scope can narrow (or eventually be deprecated). The 3 capability categories migrated in v7.0 (skills / identity / memory) were chosen because they are the load-bearing surfaces of a primary agent: without coding-delegation, AIGC routing, and persistent memory, an agent cannot function as a day-to-day driver.

### Lessons Learned

Observations from v7.0 that may inform v8.0+ planning:

- **Planning estimates benefit from early direct inspection.** The "133 files / 1.3MB" estimate in ROADMAP/STATE/REQUIREMENTS MEM-02 preceded a direct `find ... | wc -l` by ~3 phases. The verified actual (124 files / 817KB) was 37% smaller. A cheap filesystem probe at the planning stage would have caught this; instead, it propagated through docs and required a Phase 36 INGESTION-NOTE correction. Lesson: for any volume-based requirement, run the count command at `define-milestone` time, not at execution time.
- **Operator-state vs repo-commit split is load-bearing for audit.** Phase 35 (SOUL.md) and Phase 36 (USER.md + memories) both modified operator-state files outside the repo worktree. The audit trail (transformation note / ingestion note) is the **only** repo-commit artifact for those phases — without it, code review would have nothing to review. This split should be an explicit decision in any future phase that touches `~/.hermes/`.
- **Coexistence decisions are cheap to document and expensive to reverse.** Phase 34 SC #5 required a documented SUPPLEMENT-vs-REPLACE-vs-MERGE decision before any skill was migrated. Writing `COEXISTENCE.md` took ~2 minutes; reversing a wrong migration decision would have taken hours. Future skill migrations into populated categories should follow the same pattern: write the decision artifact first, migrate second.
- **"Deferred to operator" is honest scope management, not a gap.** Phases 35, 36, and 37 all surface runtime items that cannot be exercised by automated verification (live agent conversation, cloud API key, live tmux spawn). The verifier status `human_needed` correctly captures this — structural preconditions verified, runtime confirmation deferred. This is a healthier pattern than overclaiming "passed" when the runtime half was never exercised.

### Notable Cross-Phase Findings

- **124-vs-133 file count discrepancy:** Planning docs (`ROADMAP.md`, `STATE.md`, `REQUIREMENTS.md` MEM-02) all estimated "133 .md files / 1.3MB" from openclaw memory. Verified actual count at Phase 36 direct filesystem inspection: **124 files / 837,136 bytes (~817KB)** — 37% smaller than estimated. `36-01-INGESTION-NOTE.md` documents this correction. The "133 / 1.3MB" figure was a planning estimate that preceded direct `find ~/.openclaw/workspace/memory -name '*.md' | wc -l`. A doc-consistency patch is flagged but out of Phase 36 scope; operator may patch planning docs or accept the discrepancy (see Phase 36 VERIFICATION Human Verification item #4).
- **SOUL.md transformation dropped two openclaw-specific literals:** The integrated `~/.hermes/SOUL.md` correctly omits both the `GLM-4-flash` model hard-reference (replaced with generic "configured LLM" per cli-config.yaml) AND the two-agent "手/脑" framing (incompatible with hermes' single-agent model). Both drops are documented with rationale in `35-01-TRANSFORMATION-NOTE.md` §2 and §6.
- **Bidirectional related_skills graph made complete across all 6 autonomous-ai-agents skills:** Phase 34 Plan 03 wired the reverse-direction `related_skills` entries from the 4 existing skills to the 2 new ones. As a bonus opportunistic fix, a latent asymmetry was resolved: `codex` now references `opencode` (matching the pre-existing reverse link from `opencode` → `codex`). The category's discovery graph is now complete — an agent traversing from any of the 6 skills can navigate to any relevant peer.

## Forward-Looking Notes for v8.0

Suggested v8.0 priorities drawn from the deferred items documented above. These are **suggestions for v8.0 planning**, not commitments — each would require a fresh `/gsd:plan-milestone` decision. The intent is to surface the highest-leverage next moves so the operator can prioritize without re-discovering this context.

- **feishu-* skill migration + FEISHU-02 design decision:** Migrate the 4 feishu skills (feishu-doc / feishu-drive / feishu-perm / feishu-wiki) into hermes-agent. The load-bearing decision is FEISHU-02: merge into a single `feishu` skill with 4 sub-actions, or keep 4 independent skills. The decision likely depends on whether hermes grows a feishu platform plugin (which would own feishu-specific install/auth, making the skills thin wrappers) — bundle them together if so.
- **Multi-profile mechanism evaluation (AGT-01):** openclaw ran 7 separate agents with distinct profiles. Does Kai actually need 7 profiles in hermes, or is a single SOUL.md sufficient now that hermes has skill-discovery + memory + curator? If profiles are needed, the design decision is: separate `~/.hermes/profiles/<name>/` directories vs. a single SOUL.md with profile sections vs. a CLI flag. v7.0 deliberately deferred this — v8.0 should evaluate based on actual usage patterns.
- **acp-router re-evaluation (AGT-02 / ACP-01):** Could hermes serve as an ACP server instead of a client? openclaw was an ACP client of hermes; inverting that relationship might let hermes orchestrate sub-agents via ACP. Requires understanding what ACP features Kai actually uses. Probably a research-phase question before any migration.
- **Operator smoke-test outcomes from v7.0:** If any of the 4 Operator Action Items above fail at runtime (e.g., SOUL.md routing doesn't actually fire on `draw a cat` because hermes' prompt builder doesn't surface SOUL.md routing rules with enough salience), feed the failure back into v8.0 scope as a behavior-fix item. Structural preconditions passing does not guarantee runtime behavior.
- **Movie-experts subsystem continues in sibling repo (kais-movie-agent):** v7.0 deliberately does NOT touch `skills/movie-experts/` in this repo (FOUND-08 invariant preserved). v8.0 should continue to respect this — movie-experts深化 happens in the sibling repo. This repo's `skills/movie-experts/` is frozen at the v6.0 shipped state.
- **Potential cross-AI peer review of this migration report:** Per `37-CONTEXT.md` deferred items, an operator may run `/gsd:review` to have a different AI peer-review this migration report for blind spots. Not required for v7.0 closure; suggested as a v8.0-planning input.
