**Paradigm shift:** First milestone touching non-movie-experts scope. Project category broadens from "movie-experts subsystem" to "personal hermes agent platform".

**Scope:**
- Skills IN: coding-agent + tmux-agents
- Identity: single SOUL.md enhancement (no multi-profile)

## Milestone Summary

**Key Decisions:**

- **Coexistence strategy: SUPPLEMENT** — coding-agent and tmux-agents add capabilities (unified delegation + session lifecycle) that the existing 4 single-agent skills don't provide; no replacement. Bidirectional `related_skills` graph wired across all 6 autonomous-ai-agents skills.
- **SOUL.md integration: non-destructive + source-tagged** — original 513B Hermes identity preserved byte-for-byte at file head; openclaw rules added additively under `## openclaw 迁移规则 (Source: openclaw SOUL.md, migrated 2026-06-25)` header. Every rule tagged with `> **Source:** openclaw 迁移 (2026-06-25)`.
- **Memory ingestion: build tooling + defer live run** — MEM0_API_KEY not configured at v7.0 ship time; built idempotent batch_ingest.py (SHA-256 content-hash keyed) + spot_check.py with 5 topic queries. Operator-action commands documented in INGESTION-NOTE.md.
- **Trigger mode adaptation: regex → natural-language** — openclaw used regex patterns with `mcp:hermes_*` targets. Hermes-agent invocation modes are slash-commands + skill-discovery. Rules adapted to point at `skills/autonomous-ai-agents/*` (immediate), `plugins/memory/mem0/` (cognitive/memory), and `agent/curator.py` (reflection).
- **Skill file count: 124 not 133** — corrected during Phase 36 planning; planning docs (ROADMAP/REQUIREMENTS) left untouched per out-of-phase-scope rule; INGESTION-NOTE.md carries the canonical correction.

**Issues Resolved:**

- Hermes SOUL.md was reported as "0 bytes" in initial Phase 35 CONTEXT.md scout but verified to actually contain 513 bytes of Nous Research identity content. Plan correctly enforced strict additive integration (preserved byte-for-byte at file head).
- openclaw's `mcp:hermes_*` MCP tool references (`hermes_plan`/`hermes_memory_*`/`hermes_reflect`/`hermes_learn`) do not match hermes' actual MCP tool surface. Phase 35 adapted these to natural-language routing pointing at skill discovery + mem0 plugin + curator.
- Bidirectional `related_skills` graph had a latent asymmetry: `codex` did not back-reference `opencode` (despite `opencode` referencing `codex`). Phase 34-03 fixed this opportunistically while wiring the new skills.

**Issues Deferred (operator-action-handoff, NOT gaps):**

- Live mem0 ingestion (124 .md files) — blocked on MEM0_API_KEY configuration (operator handles cloud credentials)
- Live mem0 spot-check + idempotency re-test — blocked on ingested backend
- Live SOUL.md routing observation — blocked on live hermes-agent conversation
- Live coding-agent/tmux-agents skill invocation — blocked on tmux + agent CLI binaries in operator environment

These are scoped-boundary handoffs per migration-milestone design — documented in `v7.0-MIGRATION-REPORT.md` §Operator Action Items with exact commands.

**Explicitly Skipped Items (one-line rationale each):**

- `feishu-doc` / `feishu-drive` / `feishu-perm` / `feishu-wiki` — Feishu API surface too large for single milestone; deferred to v8.0+
- `acp-router` — No clean hermes analog (hermes uses gateway + platform adapters, not ACP threads)
- `models.json` migration — Operator handles provider keys manually (out of milestone flow by design)
- `agents/<name>/sessions/` runtime state — No migration value (transient session data)
- Multi-profile mechanism — v7.0 uses single SOUL.md; multi-profile deferred to v8.0+ evaluation
