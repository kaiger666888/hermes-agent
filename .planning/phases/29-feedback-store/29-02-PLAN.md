<dedup_semantics>
<!-- Exact STORE-04 behavior. Source: 29-CONTEXT.md + 29-RESEARCH Pattern 4. -->

| Condition | Action | Index Effect |
|-----------|--------|--------------|
| New sha256 (not in registry) | Append to registry; add to by_sha256 status=active | Bucket count +1; weighted_count += weight(new) |
| Same sha256 + same verdict (DUPLICATE) | REJECT — return existing record_id, do NOT write bucket file, do NOT append registry | No change |
| Same sha256 + different verdict (CORRECTION) | Append new entry to registry; mark OLD by_sha256 entry status=superseded, superseded_by=newer_id; add NEW entry status=active, supersedes=older_id | OLD verdict bucket: weighted_count -= weight(old), raw count UNCHANGED. NEW verdict bucket: count +1, weighted_count += weight(new). |

Critical nuance (RESEARCH Pattern 4 Important nuance): When CORRECTION happens, the OLDER record's bytes are still in its bucket file (we don't rewrite bucket files — they're append-only). The by_sha256[sha] entry is updated to point at the NEWER record_id with status="active", and the OLDER entry gets status="superseded". When query() scans a bucket file, it cross-references each record's sha256 against by_sha256 to determine effective status. This keeps bucket files immutable while still supporting correction semantics.

weighted_count recomputation after correction:
- Read the older bucket file; the older record is still in it (status="superseded" in by_sha256).
- _recompute_bucket_weighted_count MUST skip records whose sha256 is marked superseded in by_sha256.
- The newer bucket file gets the new record; weighted_count += weight(new).
</dedup_semantics>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 28 callers → FeedbackStore (via wrapper) | The wrapper preserves the FeedbackRecord → Path contract. Callers trust that the returned Path exists and the record is persisted. No new trust boundary — same as Plan 01. |
| Operator → rebuild-index CLI | The CLI mutates index.json. An operator running rebuild-index on a corrupted store could amplify corruption if rebuild_index has a bug. Mitigation: rebuild_index is read-only on buckets + registry (append-only files); it only rewrites index.json atomically. |
| Concurrent dedup check + write | Single-process MVP (CONTEXT.md). If two near-simultaneous writes for the same sha256 race through _handle_dedup before either updates _sha256_index, both could pass as "new" and write. Mitigation: document single-process assumption in module docstring; Plan 02's dedup catches the second one on next process init via registry scan. (RESEARCH Pitfall #3) |
