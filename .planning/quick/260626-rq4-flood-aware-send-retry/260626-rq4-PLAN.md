<objective>
Make `BasePlatformAdapter._send_with_retry()` Telegram-flood-aware. Today the helper uses exponential backoff (~2s, ~4s) which cannot out-wait Telegram's 18-27s sliding flood window — the gateway exhausts its 2 retries while the window is still active, then drops the entire response with a "Message delivery failed" notice. This fix detects flood errors, parses `retry_after` (capped at 60s, floored at 3s, default 5s), and waits that long before retrying.

Purpose: Stop losing complete agent responses to Telegram flood control. Mirror the proven pattern already shipped in `gateway/stream_consumer.py` so both layers speak the same flood vocabulary.

Output: Patched `gateway/platforms/base.py` (new constants + 2 helpers + modified `_send_with_retry`) and extended `tests/gateway/test_send_retry.py` with a `TestFloodAwareRetry` class.
</objective>

```python
_MAX_FLOOD_SUSPEND_SECONDS = 60.0
_MIN_FLOOD_SUSPEND_SECONDS = 3.0
_FLOOD_RETRY_AFTER_RE = re.compile(r"retry\s*(?:in|after)\s+(\d+)\s*seconds?", re.IGNORECASE)

def _is_flood_error(self, result) -> bool:
    err = getattr(result, "error", "") or ""
    err_lower = err.lower()
    return "flood" in err_lower or "retry after" in err_lower or "rate" in err_lower

def _extract_retry_after_seconds(self, result) -> Optional[float]:
    raw = getattr(result, "raw_response", None)
    if raw is not None:
        val = raw.get("retry_after") if isinstance(raw, dict) else getattr(raw, "retry_after", None)
        if val is not None:
            try: return float(val)
            except (TypeError, ValueError): pass
    err = getattr(result, "error", "") or ""
    m = self._FLOOD_RETRY_AFTER_RE.search(err)
    if m:
        try: return float(m.group(1))
        except (TypeError, ValueError): return None
    return None
```
