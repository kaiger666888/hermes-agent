<objective>
Fix the credential-pool bug that silently killed GLM 4-key rotation during the 2026-07-02 10:05-10:25 / 15:33-15:36 / 16:45-16:47 CST 1305 overloaded_error storms.

Root cause: the `FailoverReason.overloaded` branch in `recover_with_credential_pool` (agent/agent_runtime_helpers.py:675-690) calls `pool.mark_exhausted_and_rotate()`, which permanently marks the current key as `STATUS_EXHAUSTED` — identical to billing exhaustion. But 1305 means "model overall overloaded", NOT "this key's quota exhausted". After 4× 1305, all 4 keys are marked exhausted → `recover_with_credential_pool` returns `(False, …)` on every subsequent call → rotation becomes a no-op → user sees "model provider failed after retries".

Fix: rotate WITHOUT marking. Add `CredentialPool.rotate_to_next()` (a non-mutating rotate), and switch the overloaded branch to use it. Keep `billing`/`rate_limit`/`auth` branches unchanged.

Purpose: Stop the silent rotation death on GLM 4-key setups during provider-side capacity storms. Composes cleanly with the just-shipped quick-260702-ezx (3-strike abort + 30s/600s backoff + concurrency guard) — that plan's Task 1 explicitly noted "the existing overloaded-rotate branch is correct for multi-key users" assuming it rotates without killing keys; this fix makes that assumption true.

<implementation_note>
**Subtle behavior question for the executor:** When 4 consecutive 1305s hit a 4-key pool, what should `rotate_to_next()` do on the 4th call?

Answer: On the 4th call, the just-failed entry (call it entry D) is still "available" because we never marked it. So `_select_unlocked()` after `_current_id = None` could legitimately pick entry A (which also just failed 3 calls ago but was never marked) — meaning rotation would loop A→B→C→D→A→B→... forever.

This is INTENTIONAL and CORRECT. Here's why:
1. Each rotation advances `_current_id` to a DIFFERENT entry (because `_select_unlocked` uses strategies like fill_first which picks the lowest-priority available, and after a rotation the priorities shift).
2. After enough rotations, the upstream ezx 3-strike abort fires (3 consecutive 1305 = abort) and surfaces a clean error. The pool doesn't need to give up — the conversation loop does.
3. The "4-strike clean error" must-have truth is about: when ALL entries have been cycled through and the pool genuinely has no NEW entry to try. This is detected at the `recover_with_credential_pool` level by checking if `next_entry` is the SAME entry we just rotated away from. If so → return (False, has_retried_429). This is the "all cycled, give up" signal that lets ezx's abort message show.

**Concrete implementation:**
- `rotate_to_next(self, *, api_key_hint=None) -> Optional[PooledCredential]`:
  1. `with self._lock:`
  2. Resolve current entry (same logic as `mark_exhausted_and_rotate`: by api_key_hint if given, else `self.current() or self._select_unlocked()`).
  3. If entry is None → return None.
  4. Remember `previous_id = entry.id`.
  5. `self._current_id = None`.
  6. `next_entry = self._select_unlocked()`.
  7. If `next_entry is None` → return None (no available alternatives).
  8. If `next_entry.id == previous_id` → return None (we've cycled all the way around; no point re-trying the same key immediately).
  9. Log INFO: `"credential pool: rotating %s (overloaded — key retained) → %s"`.
  10. Return `next_entry`.

This guarantees: (a) keys are never marked exhausted on 1305; (b) rotation still happens; (c) when the pool cycles back to the same entry, recovery fails cleanly → caller surfaces error → ezx 3-strike abort takes over.

The check in step 8 is the load-bearing piece for the "4-strike clean error" must-have. Without it, the pool would loop forever.
</implementation_note>

    ```python
    def rotate_to_next(
        self,
        *,
        api_key_hint: Optional[str] = None,
    ) -> Optional[PooledCredential]:
        """Rotate to the next available credential WITHOUT marking the current one.

        Used for FailoverReason.overloaded (provider-side capacity errors like
        Zhipu GLM 1305 "model capacity exceeded").  These errors mean the MODEL
        is overloaded, not that THIS KEY's quota is exhausted — so permanently
        marking the key ``STATUS_EXHAUSTED`` (as ``mark_exhausted_and_rotate``
        does) is wrong.  Without this method, every 1305 kills one key; after
        N× 1305 (where N = pool size) all keys are exhausted and rotation
        silently dies — see 2026-07-02 incident.

        Returns the next entry to use, or None when:
        * the pool is empty, or
        * there is only one entry (no alternative to rotate to), or
        * every other entry is in exhaustion cooldown, or
        * ``_select_unlocked`` cycles back to the SAME entry we just rotated
          away from (signals "all alternatives tried" — caller should surface
          a clean error rather than loop forever).

        IMPORTANT: this method is a NO-OP on the just-rotated-away entry's
        ``last_status`` / ``last_error_*`` fields.  The key remains fully
        selectable on the next ``select()`` call.

        Contract parity with ``mark_exhausted_and_rotate``: same lock, same
        current-entry resolution (``api_key_hint`` → ``current()`` →
        ``_select_unlocked()``), same ``_current_id = None`` before
        re-selecting.  Differs ONLY in not calling ``_mark_exhausted``.
        """
        with self._lock:
            entry = None
            if api_key_hint:
                entry = next(
                    (e for e in self._entries if e.runtime_api_key == api_key_hint),
                    None,
                )
            if entry is None:
                entry = self.current() or self._select_unlocked()
            if entry is None:
                return None
            previous_id = entry.id
            _prev_label = entry.label or entry.id[:8]
            self._current_id = None
            next_entry = self._select_unlocked()
            if next_entry is None:
                logger.info(
                    "credential pool: no alternative available for overloaded "
                    "rotation from %s — recovery yields to caller",
                    _prev_label,
                )
                return None
            if next_entry.id == previous_id:
                logger.info(
                    "credential pool: overloaded rotation from %s cycled back "
                    "to same entry — all alternatives tried, recovery yields "
                    "to caller",
                    _prev_label,
                )
                return None
            _next_label = next_entry.label or next_entry.id[:8]
            logger.info(
                "credential pool: overloaded rotation %s → %s (key retained, "
                "not marked exhausted — model capacity error is not per-key)",
                _prev_label, _next_label,
            )
            return next_entry
    ```
